# Codex 使用文档

这份文档讲的是你现在对话里这个 Codex 代码助手该怎么用。它不是只会聊天的问答工具，更像是一个能读项目、改文件、跑命令、查 Git、做构建验证、重启项目的本地开发搭档。

你可以把它理解成：你负责说清楚目标和验收标准，Codex 负责在项目里查、改、测、整理结果。

## 1. Codex 适合帮你做什么

### 1.1 代码审查

你可以让 Codex 全盘检查项目，也可以只看某一块。

适合的说法：

```text
帮我全盘审查项目，重点看状态管理、积分系统、交互逻辑有没有问题。
```

```text
帮我审查刚才你改的拖拽动画，重点看会不会影响观感、性能和状态切换。
```

```text
帮我看一下这个功能有没有隐藏 bug，不要只看能不能编译。
```

Codex 审查时通常会看：

- 代码能不能编译。
- 状态流有没有冲突。
- 是否会出现重复触发、闪烁、叠层错误、内存泄漏。
- 是否破坏旧功能。
- 是否有边界情况，比如空数据、旧存档、异常输入。
- 是否需要补充测试或验证步骤。

如果你说“仔细审查”，Codex 会默认按比较严格的方式查，不只是扫一眼。

### 1.2 直接修改代码

你可以直接说“修改”，不需要每次都让 Codex 先给方案。

适合的说法：

```text
这个中断打工逻辑不合理，直接帮我改。
```

```text
移动时不要身体倾斜了，改成被手拎着的 3 帧动画。
```

```text
积分消耗太低了，帮我重新设计数值并直接改到代码里。
```

Codex 会先读项目结构，再动手改文件。修改完成后一般会做构建验证，比如：

```bash
npm.cmd run build
```

你的这个项目还额外约定了：每次修改完成后要重启桌宠项目。这个习惯很好，因为桌面应用有时候旧进程还在跑，不重启你看到的可能不是最新代码。

### 1.3 Git 分支、提交、上传 GitHub

Codex 可以帮你看分支、看提交记录、整理状态、提交、推送。

适合的说法：

```text
帮我看一下当前 git 分支和提交记录。
```

```text
把当前改动提交到新分支，不要动主分支。
```

```text
提交并上传 GitHub。
```

比较推荐的流程：

1. 先看当前分支和工作区状态。
2. 确认哪些文件是本次要提交的。
3. 写清楚提交信息。
4. 推送到远程分支。

如果你不想让 Codex 提交某些东西，要明确说：

```text
只提交代码，不要提交 dist 和日志文件。
```

```text
图像也一起提交。
```

```text
不要提交，先让我试一下。
```

### 1.4 项目架构梳理

Codex 适合把项目从“文件堆”整理成清晰的逻辑图。

你可以让它输出：

- 项目目录说明。
- 模块职责。
- 数据流。
- 状态机逻辑。
- 功能点清单。
- 后续可扩展功能。
- 哪些代码需要拆分。
- 哪些地方可能越写越乱。

适合的说法：

```text
帮我梳理整体项目逻辑架构，说明每个模块负责什么。
```

```text
为了后续加更多互动功能，帮我调整架构，别让功能越来越乱。
```

```text
帮我把当前项目功能点整理成文档。
```

### 1.5 资源处理和动画

你的桌宠项目里有大量图片资源，所以 Codex 也可以帮你做资源相关工作。

比如：

- 批量抠图。
- 检查透明背景。
- 删除无用图像。
- 生成简单帧动画。
- 把新图接入前端状态。
- 检查图片是否有黑边、白边、横竖条纹、闪烁问题。

适合的说法：

```text
帮我批量检查 public/pet/frames 里的图片透明度。
```

```text
这组图切换时在闪，帮我查原因。
```

```text
给拖拽状态做 3 帧动画，并接入代码。
```

如果你对画面要求高，最好补一句：

```text
改完后要考虑观感，不要只保证能运行。
```

这句话很有用，因为动画类改动经常不是“能跑就行”，而是要看起来顺。

## 2. Codex 的工作方式

### 2.1 它会先读代码，不应该瞎猜

比较靠谱的开发方式是：

1. 先看项目结构。
2. 搜索相关关键词。
3. 读关键文件。
4. 找到真正的状态入口。
5. 小范围修改。
6. 构建验证。
7. 必要时重启项目。
8. 告诉你改了什么。

你可以直接要求：

```text
先不要急着改，先读代码并告诉我问题在哪里。
```

也可以反过来：

```text
不用先问我，直接按你判断修改。
```

### 2.2 它会尽量保护你已有的改动

如果工作区里已经有未提交修改，Codex 不应该随便回滚。

你可以这样提醒：

```text
注意不要覆盖我现有改动。
```

```text
只改和这个功能相关的文件。
```

```text
不要 git reset。
```

默认情况下，Codex 不应该使用破坏性命令，比如强制重置、随便删除目录、覆盖用户修改。

### 2.3 它可以运行命令，但有权限边界

Codex 能运行本地命令，比如：

```bash
git status
npm.cmd run build
netstat -ano
Get-Process electron
```

但有些操作可能需要你确认，比如：

- 推送到 GitHub。
- 需要联网下载依赖。
- 删除大量文件。
- 关闭或启动外部程序。
- 写入工作区以外的目录。

如果你看到 Codex 请求权限，不一定是出错，通常是因为这个动作对系统有影响。

## 3. 怎么给 Codex 下指令更高效

### 3.1 最好说清楚“目标”

不太好的说法：

```text
这个不好，改一下。
```

更好的说法：

```text
拖拽的时候现在只是身体倾斜，我希望变成被手拎起来的动画，3 帧左右，透明背景，不要闪。
```

这样 Codex 知道你真正要的是“拖拽体验”，不是随便换一个 CSS 动画。

### 3.2 最好说清楚“验收标准”

比如：

```text
改完后要能 npm build 通过，并重启项目。
```

```text
图片必须是真透明背景，不能是模拟黑色或白色背景。
```

```text
不要只做菜单入口，还要保证积分结算合理。
```

验收标准越明确，返工越少。

### 3.3 如果你不确定方案，可以让 Codex 先思考

比如：

```text
先别改，先分析这个功能应该怎么设计。
```

```text
你先给我几个方案，说明优缺点。
```

```text
从可维护性角度讲，哪个方案更适合后续扩展？
```

这类请求适合复杂功能，比如宠物互动系统、积分经济系统、动画状态机、背包系统、任务系统。

### 3.4 如果你已经确定方向，就直接让它执行

比如：

```text
可以，按第三个方案直接做。
```

```text
不用再解释，直接修改。
```

```text
继续。
```

Codex 会沿着当前上下文继续做，不需要你重复所有背景。

## 4. 常见任务模板

### 4.1 全盘代码审查模板

```text
帮我全盘审查项目，重点看：
1. 有没有明显 bug；
2. 状态管理是否混乱；
3. 有没有重复逻辑；
4. 以后加功能会不会难维护；
5. 有没有观感或交互问题。

先列问题，再直接修你认为应该修的部分。
```

### 4.2 功能开发模板

```text
我要加一个【功能名称】。

目标：
【这里写你希望用户看到什么/能做什么】

规则：
【这里写数值、限制、状态切换等】

要求：
1. 直接修改代码；
2. 保持现有风格；
3. 构建通过；
4. 修改后重启项目；
5. 最后告诉我改了哪些文件。
```

### 4.3 UI/动画优化模板

```text
帮我优化【某个状态/动画】。

现在的问题：
【比如闪烁、太僵硬、太突兀、挡住文字】

我希望：
【比如更自然、更像被拎着、更可爱、更明显】

注意：
图片要透明背景，不要模拟背景；状态切换不要闪。
```

### 4.4 Git 提交模板

```text
帮我提交当前改动。

要求：
1. 先看 git status；
2. 确认包含哪些文件；
3. 提交信息用中文；
4. 推送到当前分支；
5. 不要切主分支。
```

### 4.5 数值系统设计模板

```text
帮我重新设计积分系统。

请考虑：
1. 获取速度；
2. 消耗速度；
3. 每日上限；
4. 防止刷分；
5. 新手体验；
6. 后续扩展空间。

设计完直接改代码，并说明改动理由。
```

## 5. 针对这个桌宠项目的使用建议

### 5.1 每次改完都重启项目

你已经定了这个规则，建议继续保持。

原因很简单：Electron 桌面应用容易出现旧进程还在跑的情况。如果不重启，你看到的问题可能是旧代码造成的。

推荐你直接说：

```text
改完记得重启项目。
```

不过你已经说过一次，Codex 后续应该默认记住这个习惯。

### 5.2 图片类任务要强调透明背景

因为你之前遇到过“模拟背景”“黑框”“条纹没扣干净”等问题，所以图片任务最好明确：

```text
必须是真透明背景，四角 alpha 要是 0，不要用黑色或白色模拟透明。
```

如果是批量抠图，可以说：

```text
只扣掉人物外面的背景，不要扣掉人物内部细节。
```

如果是动画帧，可以说：

```text
每一帧角色位置不要跳太大，不然会闪。
```

### 5.3 动画任务不能只看单帧

单张图好看，不代表动画好看。

动画类任务最好检查：

- 每帧主体大小是否一致。
- 角色中心点是否跳动太大。
- 透明边界是否一致。
- 帧率是否太快。
- 切换状态时是否闪白、闪黑、出现占位文字。
- UI 元素是否挡住角色。

你可以直接让 Codex 做这种检查：

```text
不要只检查图片，帮我从动画连续性和观感上审查。
```

### 5.4 数值系统要防止“立即收益”

你之前发现“1 小时打工立即中断拿 14 分”就是典型数值漏洞。

以后做数值系统时，最好要求 Codex 检查：

- 有没有固定保底导致刷分。
- 有没有冷却时间。
- 有没有每日上限。
- 取消、中断、失败时的收益是否合理。
- 长任务和短任务的收益比例是否合理。
- 玩家会不会找到明显套利方式。

可以这样说：

```text
帮我从玩家刷分的角度反推这个系统有没有漏洞。
```

### 5.5 菜单入口要符合使用场景

右键菜单不是越多越好。

比如“中断打工”放顶层就不自然，放进“打工”子菜单更合理。

以后新增功能可以考虑：

- 高频操作放顶层。
- 状态相关操作放对应子菜单。
- 只有当前状态可用的操作才显示或启用。
- 危险操作要有确认或不放太显眼。

## 6. Codex 修改代码后的结果你怎么看

Codex 最后一般会告诉你：

- 改了哪些文件。
- 改了什么逻辑。
- 跑了什么验证。
- 有没有重启项目。
- 还有什么风险。

你看结果时，可以重点看这几件事：

1. 是否真的解决了你说的问题。
2. 是否引入了新问题。
3. 是否改动范围过大。
4. 是否有没提交的文件。
5. 是否有生成物、缓存、日志混进来。

如果你觉得不对，可以直接说：

```text
这个方案不行，问题是……
```

```text
你再审查你自己写的代码，自我纠错。
```

```text
这次从观感角度重新想，不要只看功能。
```

Codex 会根据新反馈继续改。

## 7. Codex 的局限

### 7.1 它会犯错

Codex 能读代码、改代码，但不代表每次判断都完美。尤其是图片生成、动画观感、复杂状态切换这种任务，第一次结果可能需要调整。

所以你让它“自查”“再次审查”“从观感角度思考”是对的。

### 7.2 它不一定能直接看到你屏幕上的真实效果

Codex 可以通过文件、构建、截图、图片检查来判断问题，但你实际看到的桌面宠物效果可能还有窗口透明、系统缩放、显卡渲染等因素。

如果你看到问题，最好描述得具体一点：

```text
拖动的时候手太大了。
```

```text
切换瞬间闪了一下黑边。
```

```text
右上角还有一条没扣干净的线。
```

越具体，越好修。

### 7.3 它不能替你判断所有审美偏好

比如“可爱一点”“自然一点”“别太突兀”这种要求，Codex 可以按常规审美优化，但最终还是要看你的感觉。

你可以用更具体的描述来提高命中率：

```text
动作幅度小一点，像轻轻晃，不要像左右跳。
```

```text
手不要挡住脸，放在帽子上方。
```

```text
动画慢一点，不要太急。
```

## 8. 推荐的协作节奏

### 8.1 小功能

小功能可以直接让 Codex 改：

```text
帮我把喂食积分消耗调高一点，直接改。
```

然后你试效果，不满意再继续调。

### 8.2 中等功能

中等功能建议先让 Codex 想方案：

```text
我想加一个任务系统，你先设计一下结构和功能点，不要急着写代码。
```

确认后再说：

```text
可以，按这个方案做。
```

### 8.3 大功能

大功能最好拆阶段：

1. 先梳理现有架构。
2. 再设计数据结构。
3. 再做最小可用版本。
4. 再补 UI 和动画。
5. 最后审查和提交。

比如：

```text
我要加背包系统，先帮我梳理现有积分、喂食、状态机怎么接。
```

```text
先做最小版本：能买食物、能存在背包、能使用。
```

```text
再优化 UI 和动画。
```

这种节奏更稳，不容易把项目写乱。

## 10. 最推荐的使用方式

最推荐的方式其实很简单：

1. 你先说清楚你想要的效果。
2. Codex 先读代码，不乱猜。
3. Codex 做小范围修改。
4. Codex 构建验证并重启。
5. 你实际试一下。
6. 你把观感问题说出来。
7. Codex 再继续调。

这个循环比一口气做一大堆功能靠谱得多。

尤其是你的桌宠项目，很多问题不是“代码对不对”，而是“看起来顺不顺”“状态切换自然不自然”“图片有没有透明边缘问题”。这类项目就应该多试、多审、多调。
